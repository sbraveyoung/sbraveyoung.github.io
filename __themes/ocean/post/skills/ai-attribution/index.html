Skill · AI 标识三档约定

何时用：写完一篇文章决定要不要标 AI / 标哪一档时；或者别人来问"你这博客 AI 标签是啥意思"时甩链接给他。

解决什么：把"哪种程度的 AI 参与对应哪种 badge"明文化，跟读者建立预期，跟自己建立纪律。

三档来历

借鉴的是 IPTC digitalSourceType 这个媒体溯源词表：

  • trainedAlgorithmicMedia — 纯 AI 生成
  • compositeWithTrainedAlgorithmicMedia — 人 + AI 混合
  • algorithmicallyEnhancedMedia — 人为主，AI 增强

翻译成日常说法：生成 / 协作 / 校对——刚好对应博客上的 AI 生成 / AI 辅助 / AI 校对。再加上"模型署名"这个常见用法，一共四种：

front-matter ai: badge 含义
true / yes / 1 / on / assisted 🤖 AI 辅助 我主笔，AI 帮我顺手（默认档）
generated / wrote / written 🤖 AI 生成 AI 主笔，我审稿 / 改 / 校
edited / reviewed / polished 🤖 AI 校对 我写完，AI 帮我润色 / 校对错字
claude / gpt-4o / <任何字符串> 🤖 <原样> 模型署名，原样显示
false / no / 0 / off / 留空 （无 badge） 我自己写的，AI 没碰

值大小写不敏感（True / Generated / EDITED 都能识别）。

我自己的判断准则

场景 标什么
我用 IDE 跟 Copilot / Claude Code 一起写代码并解释，文章完全是我自己敲的 （不标）
我让 AI 帮我列了大纲，正文我写的 ai: true
我跟 AI 反复对话，最终 AI 写大段文字我顺手改了一些 ai: generated
我写完整篇，让 AI 检查错字 / 改通顺 ai: edited
我用 Claude 把一篇英文文章翻译过来 ai: claude

不绝对，但别假装独立写作——这是核心。AI 帮过忙就标，标得越具体越好。

怎么写

front-matter 里加一行：

---
title: ...
ai: true       # 或 generated / edited / <model name>
---

仅此而已。badge 自动出现在文章顶部 meta 行里。

不标的反而更可疑

我的潜在底线：写得越流畅、越规整的内容越要主动标。读者看到一篇行文工整、术语精确、转折流畅的技术文章，会下意识觉得"作者写得很认真"。如果实际上是 AI 帮了一大半的忙，不标 = 误导。

反过来，写得磕磕巴巴、有典型口头禅的内容（"xxx"、"哎"、"我也不知道为啥"），反而是人味的标志，不标也基于读者直觉自洽。

同行做法

  • 少数派 (sspai)：有 "AI 辅助" 的内容标签。
  • Medium：作者发布时有 "AI usage" 选项（Assisted / Generated）。
  • Nature 期刊：要求论文里使用 generative AI 必须在 methods 或 acknowledgments 中说明。
  • AP Stylebook：建议在新闻报道里实质性使用 AI 时披露。
  • Anthropic Claude Code：commit 信息里加 Co-Authored-By: Claude <noreply@anthropic.com> trailer。
  • C2PA / Adobe / OpenAI：在 DALL-E 等产物里嵌入加密的 provenance manifest。

我这个三档就是把上面"披露要分级"的精神放到博客上了——选择粒度，自己决定。

一处约定，全栈生效

AILabel() 函数在 sbraveyoung/gobog/src/article/article.go。改这个枚举 = 影响所有主题、所有 badge 渲染、atom feed 是否带这个标签——全栈一处。
